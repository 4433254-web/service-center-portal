## Блок 2: Архитектурные изменения (мультиточки)

### 2.1 Новая сущность: ServiceLocation (точка / сервис)
Таблица `service_locations`:
- id, name, address, phone, email
- created_by (admin), is_active
- Пользователи привязаны к точке через user_locations

Правила:
- Admin создаёт точки и назначает менеджеров
- Менеджер управляет только своей точкой (мастера, приёмщики)
- Заказ принадлежит точке где принят
- Заказ можно ПЕРЕДАТЬ в другую точку (transfer)

### 2.2 Изменения в заказе
Поле `location_id` — где принят  
Поле `assigned_location_id` — где выполняется (может отличаться)  
Поле `master_user_id` — назначается при создании или позже  
Поле `transferred_from_location_id` — если передан

### 2.3 Новые статусные переходы
Добавить прямые переходы:
- accepted → in_progress (без диагностики — мелкий ремонт)
- accepted → ready (сразу выдаём — программы, чистка)
- in_progress → ready

Полная матрица:
```
accepted → [in_diagnostics, in_progress, ready, cancelled]
in_diagnostics → [waiting_approval, in_progress, cancelled]
waiting_approval → [in_progress, cancelled]
in_progress → [ready, cancelled]
ready → [issued]
```

### 2.4 Фотографии к заказу
Таблица `order_photos` (через files с entity_type='order_photo'):
- Фото при приёмке (этап accepted)
- Фото в процессе ремонта (этап in_progress)
- Фото после ремонта (этап ready)
- Комментарий к фото
- Автор и время

Frontend: загрузка фото на каждом этапе карточки заказа

## Блок 3: Роли и права (мультиточки)

| Роль | Создаёт | Видит | Управляет |
|------|---------|-------|-----------|
| admin | Всё, все точки | Всё | Всё |
| manager | Пользователей своей точки | Заказы своей точки | Свою точку |
| receiver | Заказы, клиентов, устройства | Заказы своей точки | — |
| master | — | Только назначенные заказы | Статус своих заказов |

Один человек может быть receiver И master одновременно (роли не исключают друг друга).

## Блок 5: Мобильные приложения — стратегия

### Технология: React Native (Expo)
Причины:
- Один кодовой базой → iOS + Android
- Команда уже знает React (Next.js)
- Expo упрощает сборку и публикацию
- Переиспользуем api.ts, helpers.ts, логику

### Что будет в мобильном приложении

**Для приёмщика:**
- Список заказов (фильтр по точке)
- Создание заказа (3 шага: клиент → устройство → заказ)
- Сканирование QR для быстрого открытия заказа
- Фото при приёмке (камера телефона)
- Квитанция (PDF preview)

**Для мастера:**
- Список назначенных заказов
- Карточка заказа со статусами
- Добавление фото ремонта
- Комментарии

**Для менеджера:**
- Дашборд точки
- Статистика
- Просмотр заказов

**Для admin:**
- Полный доступ
- Управление точками и пользователями

### Архитектура мобильного приложения
```
mobile/
├── app/               # Expo Router (файловый роутинг)
│   ├── (auth)/
│   │   └── login.tsx
│   ├── (tabs)/
│   │   ├── orders/
│   │   ├── clients/
│   │   └── dashboard/
│   └── orders/[id].tsx
├── components/
├── lib/
│   ├── api.ts         # Тот же API клиент
│   └── auth.ts
└── app.json
```

### Публикация
- iOS: App Store (нужен Apple Developer $99/год) или TestFlight
- Android: Google Play ($25 разовый взнос) или APK напрямую
- Для внутреннего использования: Expo Go (бесплатно, моментально)
